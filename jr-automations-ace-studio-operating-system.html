Case Study

JR Automations to Ace Studio Operating System

A business-first operating layer for JR Ace of Fades that evolved from shortcut-based capture into a dashboard-backed review and decision workflow.

Type
Business system / workflow engineering
Core tools
Google Sheets, Apple Shortcuts, Apps Script, dashboard design
Role fit
Automation, operations systems, knowledge and workflow design
Back to portfolio GitHub

Problem

Useful business information was spread across memory, notes, app checks, and repeated admin work. Money, chair rent, stock, content ideas, and weekly priorities were easy to lose or delay because there was no simple operating layer that could be used quickly from the phone and reviewed later in one place.

Business System

The project started from a practical business need around JR Ace of Fades and grew into the broader Ace Studio operating system. The point was to create one structure for money tracking, chair-rent logging, stock visibility, content capture, and weekly review, while keeping secondary personal workflows available without letting them take over the core business use case.

Automation Layer

Apple Shortcuts became the capture layer. Repeated actions such as income logging, expense logging, chair rent, stock checks, content ideas, weekly review, and brain dumps were turned into structured input flows. Those Shortcut submissions post JSON into a Google Apps Script web app, which routes records into the correct Google Sheet tabs.

Dashboard And Decision Layer

Google Sheets acts as the durable data layer, while Apps Script provides the review layer. The dashboard was redesigned around a command-centre view with money, stock, weekly priority, recent activity, and public Barbr context. The important technical point is that the reporting layer improved without breaking the existing Shortcut POST contract.

Why It Matters

This is one of the clearest examples of how I work: start with repeated operational friction, build a practical input system, preserve structure, and make the output useful for decisions rather than passive record keeping. It is a stronger workflow-engineering story than a generic automation demo.

What It Shows

  • Designing across input, storage, processing, and review layers
  • Keeping automation grounded in real business use rather than novelty
  • Turning scattered admin work into a reusable operating system
  • Evolving a rough idea into a clearer dashboard-backed workflow
